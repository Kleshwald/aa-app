# Дизайн-инструкция для Claude Code — кабинет страхового агента

> Дизайнер проекта — **Баха** (субагент `.claude/agents/baha.md`). Этот файл — его «закон».
> Claude Code обязан следовать ему при **любой** работе с интерфейсом: новые экраны, правки, ревью, генерация компонентов.
> Принцип простой: **консистентность важнее креатива.** Язык дизайна уже определён (1С 8.5). Задача — дисциплинированно работать внутри него, а не изобретать новый стиль на каждом экране.

---

## 0. Контекст продукта (читать первым)

- **Что это:** агрегатор — платформа для страховых агентов, оформление полисов (ОСАГО и др.).
- **Кто пользователь:** профессиональные агенты, преимущественно женщины 45+, небольшие города. Заполняют сложные формы по многу раз в день.
- **Что для них «хорошо»:** скорость, ясность, отсутствие ошибок, ничего не нужно гадать. НЕ «вау-анимация».
- **Язык дизайна:** 1С:Предприятие 8.5 — воздушные отступы, карточки, скругления, векторные иконки, современная типографика. Пока — **только светлая тема** (но цвета задаются через токены, чтобы тёмная добавлялась позже без переписывания).
- **Стек:** Angular (актуальная версия).

**Главный фильтр для любого решения:** «Сможет ли агент 50 лет заполнить это быстро и без ошибок?» Если решение делает экран красивее на скриншоте, но медленнее в работе — оно отклоняется.

---

## 1. Незыблемые правила (hard rules — нарушать нельзя)

1. **Подписи полей — всегда сверху и постоянные.** Никаких floating-label и никаких подписей-плейсхолдеров. Подпись не должна исчезать, уменьшаться или менять контраст при вводе.
2. **Плейсхолдер** используется только как **пример формата** («например, А123ВС777»), никогда как название поля.
3. **Под каждым полем — стабильный слот** для подсказки/ошибки. Layout не должен «прыгать» при появлении ошибки.
4. **Ошибка показывается текстом + цветом + иконкой**, никогда только цветом рамки. Текст ошибки говорит, что не так и как починить.
5. **Размер вводимого текста ≥ 16px**, высота интерактивных полей и кнопок **≥ 44px** (читаемость для 45+ и отсутствие авто-зума на iOS).
6. **Контраст:** основной текст ≥ 4.5:1, крупный/второстепенный ≥ 3:1. Нельзя жертвовать контрастом ради «эстетики приглушённости».
7. **Никакого хардкода.** Цвета, отступы, радиусы, размеры шрифта — только из токенов (см. §2). Инлайн-хексы и «магические числа» в шаблонах запрещены.
8. **Одинаковые элементы выглядят одинаково на всех экранах.** Если где-то понадобился новый вид кнопки/поля — это добавляется как вариант в общий компонент, а не верстается заново.

---

## 2. Дизайн-токены (источник консистентности)

Всё оформление строится из ограниченного набора токенов. Значения ниже — стартовые, их можно подогнать под бренд, но **набор шкал менять нельзя** (никаких произвольных «23px», «#3a7bd5» по месту).

**Отступы (4-based):** `4, 8, 12, 16, 24, 32, 48`. Между логическими блоками — 24/32, внутри блока — 8/12/16.

**Типографика (роли, не размеры по вкусу):**
- Заголовок секции: 17px / 650
- Подзаголовок: 15px / 600
- Подпись поля (label): 13–14px / 560
- Тело / вводимый текст: 16px / 450–550
- Подсказка/caption: 12.5–13px / 450, цвет `muted`

**Радиусы:** поля и кнопки `10px`, карточки `16px`. Один масштаб на весь продукт.

**Цвета — семантические токены** (не «синий», а роль):
- `bg` — фон страницы
- `card` — фон карточки
- `ink` — основной текст
- `muted` — второстепенный текст / подсказки
- `line` — границы и разделители
- `accent` — основное действие, активное состояние
- `success / warning / error` — статусы

Все цвета — через CSS variables / SCSS-токены, чтобы добавить тёмную тему = переопределить токены, а не править компоненты.

> В нашем репо токены живут в `src/styles/tokens.scss` (имена `--gray-*`, `--header-bg`, `--success-*`, `--radius-*`, `--text-*`). Это и есть «источник истины» — новые значения добавляем туда, а не хардкодим по месту.

**Тени:** максимум 2 уровня (карточка, поднятый элемент). Не плодить.

---

## 3. Компоненты: переиспользовать, не копипастить

- **Field-компонент — единственный способ делать поля ввода.** Он включает: label сверху, контрол, слот hint, слот error, все состояния (default / focus / filled / error / disabled). Верстать поле «руками» в шаблоне запрещено.
- **Кнопки** — один компонент с вариантами (primary / secondary) и размерами. Primary-действие на экране одно.
- **Карточка-секция** — один компонент (заголовок + контент). Группировка формы строится из него.
- Прежде чем создавать новый компонент — проверь, нет ли существующего, который покрывает кейс вариантом. Новый компонент создаётся, только если это новая сущность, а не новый внешний вид старой.

---

## 4. Формы (ядро продукта — максимум внимания)

- **Группируй поля в секции/карточки** по смыслу (ТС, водители, период, контакты).
- **Короткие парные поля — в две колонки** (госномер + VIN, год + мощность). Длинные/важные — в одну колонку на всю ширину.
- **Длинные формы (ОСАГО) — разбивай на шаги (stepper)** с видимым прогрессом «Шаг N из M».
- **Маски и автоформат** для типовых полей: госномер, VIN (17 символов), даты, телефон. Подсказка формата — в hint под полем.
- **Логичный порядок и табуляция:** Tab ведёт по полям в естественном порядке заполнения.
- **Все состояния поля определены и единообразны:** покой, фокус (видимая подсветка), заполнено, ошибка, заблокировано.
- **Кнопка действия называется тем, что делает:** «Рассчитать», «Оформить полис», не «Отправить».
- Не прячь обязательность и подсказки за наведением — пользователь должен видеть их сразу.

---

## 5. Доступность (для аудитории 45+ — это требование, а не опция)

- Каждый `label` программно связан с `input` (`for`/`id`).
- Видимый фокус на всех интерактивных элементах (не убирать `outline` без замены).
- Тач-цели ≥ 44px.
- Информация никогда не передаётся только цветом — всегда дублируется текстом/иконкой.
- Интерфейс не ломается при зуме браузера до 150–200% и при крупном системном шрифте.
- Уважать `prefers-reduced-motion`.

---

## 6. Angular-специфика

- Если используется **Angular Material** — его дефолтный floating label **не использовать**. Обернуть `mat-form-field` в собственный Field-компонент с label сверху (или отказаться от `mat-form-field` в пользу своего поля). Дефолт фреймворка ≠ решение продукта.
- В нашем репо UI на **Taiga UI 5**. Подписи делаем **сверху** (`<span>` над `<tui-textfield>`), а не плавающий `tuiLabel`. Это сознательное отступление от дефолта Taiga ради §1.
- Стили — из единого источника токенов (CSS variables / SCSS), **не инлайн в шаблонах**.
- Переиспользуемые презентационные компоненты, `ChangeDetectionStrategy.OnPush` где возможно. Отзывчивость интерфейса — часть «премиальности»; лаги её убивают сильнее, чем любой визуал.
- **Каверза пайплайна:** если экран рендерится в картинку через `wkhtmltoimage` (для презентаций/превью) — там `display:grid` не поддерживается, колонки делать на `flex` с `flex:1`. В самом Angular-приложении grid работает нормально — ограничение касается только рендера картинок, не рантайма.

## 7. Чеклист ревью — «на что смотреть, когда запускаешь дизайнера»

Прогоняй по любому новому/изменённому экрану. Каждый пункт — да/нет; «нет» = правка до мёржа.

1. Подписи полей сверху, постоянные, нормального размера и контраста? (floating/placeholder-label отсутствует)
2. У каждого поля есть стабильный слот под hint и ошибку, layout не прыгает?
3. Ошибки — текстом + иконкой + цветом, с понятной формулировкой?
4. Вводимый текст ≥16px, поля/кнопки ≥44px?
5. Контраст текста проходит порог (основной ≥4.5:1)?
6. Все цвета/отступы/радиусы/шрифты — из токенов, без хардкода?
7. Одинаковые элементы выглядят одинаково с другими экранами?
8. Поля собраны через общий Field-компонент, кнопки/карточки — через общие компоненты?
9. Длинная форма разбита на секции/шаги, прогресс виден?
10. Короткие парные поля — в колонки, длинные — на всю ширину?
11. Маски/автоформат и подсказки формата на месте (VIN, госномер, даты)?
12. Все состояния полей определены (focus/filled/error/disabled)?
13. Видимый фокус, тач-цели, связь label↔input — на месте?
14. Не ломается при зуме 200% и крупном шрифте?
15. Названия кнопок описывают действие, копирайт ясный и в одном стиле?
16. Воздуха достаточно (отступы по шкале), нет визуальной тесноты?

---

## 8. Анти-паттерны (красный флаг — отклонять сразу)

- Floating label или placeholder вместо подписи.
- Ошибка, обозначенная только красной рамкой.
- Текст ввода <16px, тесные мелкие поля.
- Хардкод цветов/отступов в шаблонах, «магические числа».
- Один и тот же элемент выглядит по-разному на разных экранах.
- Низкоконтрастный «приглушённый» текст ради эстетики.
- Декоративная анимация, которая замедляет заполнение.
- Креатив ради креатива: нестандартный паттерн там, где работает стандартный.
