# Система сообщений (поддержка + куратор)

Снимок решений на 2026-06-13. Это хендофф: чтобы в следующей сессии продолжить без повторного обсуждения.

## Цель

Агент пишет в поддержку из кабинета, наша сторона отвечает и решает вопрос.
Три роли — три интерфейса:

| Роль | Интерфейс |
|------|-----------|
| **Агент** | раздел «Сообщения» в кабинете — **наш чат** (см. ниже) |
| **Поддержка** (~3 чел.) | пульт оператора выбранного бэкенда |
| **Куратор** (~15 чел.) | отвечает в той же теме + позже дашборд-обзор |

## ✅ Что РЕШЕНО и СДЕЛАНО (прототип)

- **Агентский UI — наш собственный чат, НЕ сторонний виджет.** Виджет Chatwoot был протестирован и **отклонён** (шумный, чужой, угол экрана). Реализован полноэкранный чат в нашей дизайн-системе: [messages.page.ts](../src/app/features/messages/messages.page.ts).
- **Модель диалога: одна тема, две роли.** В одной ленте отвечают и Поддержка, и Куратор; каждый ответ подписан (имя · роль) и с аватаром-инициалами. Подтверждено пользователем. Альтернатива (раздельные вкладки / много тем) — отклонена как сложная для аудитории.
- **Непрочитанные** — бейдж на пункте «Сообщения» (не отдельный колокольчик).
- **Мок-бэкенд** [chat.service.ts](../src/app/core/services/chat.service.ts): приветствия, контекстные авто-ответы по ключевым словам, optimistic send, localStorage. **Это сервис-граница** — под неё позже встанет реальный адаптер (REST + realtime), сам UI не изменится.

## ⏳ Что НЕ решено — выбор бэкенда (главное на следующую сессию)

**Критично: 152-ФЗ → данные только в РФ.** Поэтому **Chatwoot Cloud (app.chatwoot.com) для прода отклонён** (хостинг США/ЕС). Текущий токен в истории — это была throwaway-песочница; в коде Chatwoot уже удалён.

Реальная развилка — РФ-решение. Сравнение по нашим осям (152-ФЗ + «всё нужно» + МАКС-потом + наша боль «чистый чат» + звонки-потом):

| Платформа | Хостинг/152 | МАКС | Ops (метки/назначение/отчёты) | Свой UI | Профиль |
|-----------|-------------|------|------------------------------|---------|---------|
| **Webim** | on-premise + облако РФ | через Custom Channel API | да (банки) | Mobile SDK; web-API — **уточнить** | макс. контроль, «суверенный» |
| **edna** | облако РФ | **нативно** (с 2025) | да | через API — уточнить | нативный МАКС, меньше риска |
| **Carrot quest** | РФ, реестр ПО, 242-ФЗ | **нативно** | средне (маркетинг-чат, Intercom-клон) | API+SDK | лучший комплаенс; но виджет — тот же класс, что Chatwoot |
| **Usedesk** | РФ | через интеграции | сильный тикетинг | API | если приоритет — операционка |
| **Chat2Desk** | РФ, открытый API | да | средне (мессенджер-агрегатор) | API | если поддержка в основном в мессенджерах |
| **Bitrix24** | коробка + облако РФ | через коннектор (Wazzup/Radist) | да + CRM | imconnector; **REST Открытых линий слаб для дашборда** | только если нужен общий CRM/контакт-центр |

**Предварительный фаворит-набор на демо:** **Webim** (под «свой чистый чат + суверенность») + **edna или Carrot quest** (страховка по нативному МАКС).

**Что проверить на демо у каждого (4 пункта):**
1. Хостинг on-premise/облако-РФ + бумаги 152-ФЗ.
2. МАКС: нативно или через custom-API; что готово сейчас.
3. **Web client API/SDK** — встроить чат в наш Angular под нашу типографику (не их виджет). ← для нас ключевое.
4. API на диалоги/отчёты для дашборда куратора + цена на ~3 операторов / ~500 агентов.

## МАКС (мессенджер) — будущий канал

- У МАКС **открытый Bot API** (токен через business.max.ru, верификация юрлица РФ).
- Нативно поддерживают **edna, Carrot quest, Chat2Desk**; у Bitrix — через коннектор; у Webim — через Custom Channel API.
- **Требование на чек-лист:** объединяет ли вендор треды одного агента из разных каналов (кабинет + МАКС) в **одну тему по идентичности**. Иначе история разорвётся (в МАКС — своё, в кабинете — своё; оператор видит обе под одной карточкой, но агент — нет).

## Звонки — будущая возможность

3 формата, добавляются поверх выбранного бэкенда как ещё один канал:
1. **Телефон (PSTN)** → РФ-ВАТС: Mango Office / Telfin / UIS / Zadarma (запись в РФ, готовые коннекторы к helpdesk).
2. **Звонок из кабинета (WebRTC)** → **Voximplant** (РФ, Web SDK). Самый вероятный сценарий: кнопка «Позвонить в поддержку» прямо в кабинете.
3. **Внутри МАКС** → у МАКС есть аудио/видео; бизнес-звонки уточнять по мере развития.
- Если звонки станут ядром и едиными с чатом → **Bitrix24** (нативная телефония) поднимается в шортлисте.
- Запись разговоров = перс. данные → хранение в РФ (все названные провайдеры закрывают).

## Дашборд куратора — ПОТОМ (явно отложено)

Сначала рабочий прототип чата агент↔поддержка, потом дашборд. Дашборд = кастомный Angular на REST API выбранного бэкенда (KPI + таблица обращений + фильтры + детали read-only). Для прототипа — мокать как остальное.

## Следующие шаги (порядок)

1. **Выбрать бэкенд** (демо Webim + edna/Carrot quest по 4 пунктам) — ДО стройки реального адаптера (у всех разный API).
2. До выбора — прототип живёт на моке `ChatService` (UX независим от бэкенда).
3. После выбора — реальный адаптер за `ChatService`-границей (REST + realtime), UI не трогаем.
4. Дашборд куратора.
5. Позже — звонки (Voximplant/ВАТС) и канал МАКС.
