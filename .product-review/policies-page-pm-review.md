# PM-ревью страницы «Мои клиенты» / «Мои полисы»

> Контекст: Прототип V1 Agent Academy. Аудитория — Елена 48 (Минусинск), Татьяна 56 (Ермаковский р-н), Светлана 67 (Шелопугинский р-н), Дмитрий 34 (Усть-Илимск). Десктоп 1366×768, нестабильный 3G, привычка к Госуслугам и Сбер Онлайн.
> Источник правды: PROJECT_BRIEF.md (раздел 4 «Мои полисы»), USER_PROFILE.md, DESIGN_SYSTEM.md, api-contract.yaml (`/policies`), `policies.fixture.ts`.

## TL;DR

1. **Раздел называется «Мои полисы»**, не «Мои клиенты». Текущая 1С путает агентов, и API/моки уже оперируют сущностью `Policy`. Отдельный раздел «Клиенты» в V1 не делаем — это второй продукт, у которого пока нет JTBD.
2. **Главный сигнал на странице — «дни до окончания» с цветовым кодом**, не «Цена». Деньги агента — пролонгации (см. PROJECT_BRIEF цели). Сейчас этой колонки нет — это главная функциональная дыра.
3. **Убираем колонки «ИКП» и «Куратор»** — они одинаковы для всех строк («мои полисы»), это чистый шум, отнимающий 20% ширины при 1366×768.
4. **Возвращаем колонку «Комиссия»** рядом с «Премией». Это рабочий инструмент агента, скрывать комиссию — антипаттерн для аудитории, чей JTBD = «заработать».
5. **Пресет-табы над таблицей**: «Требуют действия / Истекают в 30 дней / Активные / Все». Это убирает необходимость крутить фильтры — открыл и сразу видишь, что делать сегодня (как «Платежи на сегодня» в Сбер Онлайн).

## Название раздела

**Выбор: «Мои полисы»** (route `/policies`, заголовок страницы «Мои полисы»).

**Почему:**

- API и моки уже оперируют `Policy` — один клиент может иметь несколько полисов (ОСАГО + ипотека). Если назвать раздел «Клиенты», но показывать строки полисов, агент видит дублирование ФИО и теряет ментальную модель. В 1С эта же путаница есть и сейчас вызывает жалобу «непонятно».
- В USER_PROFILE.md ментальная модель агента построена вокруг **сделок**: «оформил полис → получил комиссию → через год пролонгировал». Полис — единица учёта. Клиент — атрибут полиса.
- В Госуслугах используется аналогичный паттерн: «Документы», «Платежи», «Уведомления» — раздел называется по сущности, которую видишь в списке. Не «Мои дела», а «Заявления».
- Раздел «Клиенты» (как отдельная карточка человека со всеми его полисами и историей) — это **V2**, когда появятся CRM-сценарии (звонки, заметки, повторные продажи). В прототипе V1 такого сценария нет (см. PROJECT_BRIEF «5 ключевых сценариев»).

**В sidebar:** иконка «портфель» / «документ» + подпись **«Мои полисы»** (иконка с подписью — обязательное правило из USER_PROFILE).

## Что должно быть на странице (приоритезированный список)

### P0 — must-have, без этого больно

- **Колонка «Истекает / дни до окончания»** с цветовым кодом (>30 дней — нейтрально, 15–30 — warning, <15 — error). JTBD: «сегодня утром открыл систему — увидел, что и кого пролонгировать». Без этой колонки агент теряет деньги.
- **Пресет-таб «Истекают в 30 дней»** (по умолчанию агент сначала видит «Все», но таб с цифрой-бейджем — например «Истекают (12)» — должен бросаться в глаза).
- **Поиск по ФИО клиента, номеру полиса, госномеру** — это уже в API (`search` параметр). На странице — крупное поле поиска **слева сверху над таблицей**, не в правом углу. Поиск — самое частое действие на этой странице (звонит клиент → агент ищет его полис).
- **Действие «Пролонгировать»** прямо в строке (кнопка/иконка в последней колонке) — без необходимости открывать детали. API `/policies/{id}/prolong` готов.
- **Sticky header таблицы** при скролле (явно прописано в DESIGN_SYSTEM.md).
- **Loading skeleton, empty state, error state** (DESIGN_SYSTEM требование, и для нестабильного 3G критично).
- **Колонки** в порядке от важного к второстепенному: Клиент → Объект → Истекает → Премия → Комиссия → Статус → СК → Действия.

### P1 — нужно для V1 прототипа

- **Пресет-табы**: «Все / Истекают (N) / Активные / На оформлении / Истёкшие». Цифры в скобках — счётчики (`meta.total` из API).
- **Фильтры в видимой панели** (не за дропдауном «Фильтры»): тип продукта, СК, период. Применяется сразу при выборе.
- **Сортировка по клику на заголовок колонки** (стрелка вверх/вниз) — для endDate, premium, commission, createdAt. API уже поддерживает `sortBy`.
- **Пагинация** (страница 1 из N), 20 строк на странице. Infinite scroll **не подходит** — старый ноутбук + 3G + аудитория, привыкшая к страницам в Госуслугах.
- **Подсветка строки при hover** (gray-100), клик по строке = открытие детали.
- **Бейдж статуса в строке** (зелёный/серый/красный кружок + текст). Не просто цветной текст — иконка-метка для дальтонизма (см. WCAG в USER_PROFILE).
- **Кнопка «Оформить новый полис»** справа сверху над таблицей — самое заметное место (CTA-вход в расчёт ОСАГО).

### P2 — можно отложить

- Массовые действия (чекбоксы + «Выгрузить выбранные», «Массовая пролонгация»).
- Сохранённые фильтры («Мои фильтры»).
- Экспорт в Excel (агенты иногда хотят, но в прототипе это имитировать необязательно).
- Виды отображения (карточки vs таблица).
- Группировка по клиенту (это путь к разделу «Клиенты» в V2).
- Заметки агента к полису.

## Колонки таблицы

**Ограничение:** 1366×768, sidebar ~80–240px (зависит от свёрнутости), полезная ширина ~1100–1280px. При высоте строки 56px и шрифте 16px реально читаемо **8–9 колонок** (если ужимать — теряем DESIGN_SYSTEM-требование к высоте/шрифту).

**Беру 8 колонок:**

| # | Колонка | Источник | Ширина | Зачем |
|---|---------|----------|--------|-------|
| 1 | **Клиент** | `clientName` | ~200px | Главный идентификатор. Двустрочно: ФИО сверху, телефон серым снизу. |
| 2 | **Объект страхования** | `vehicleBrand + model` + госномер / адрес ипотеки / ФИО застрахованного | ~180px | Уточняет, какой именно полис у клиента (если их несколько). |
| 3 | **Продукт** | `type` (ОСАГО / НС / Ипотека / Антиклещ) | ~110px | Чип-бейдж с иконкой продукта. |
| 4 | **Истекает** | `endDate` + вычисляемые дни | ~140px | Двустрочно: дата сверху, «осталось 12 дней» серым/красным снизу. **Самая важная колонка.** |
| 5 | **Премия** | `premium` | ~110px, right-align | Что заплатил клиент. |
| 6 | **Комиссия** | `commission` | ~110px, right-align | **Что заработал агент.** Должна быть видна, не скрыта. |
| 7 | **СК** | `insuranceCompanyName` | ~120px | Логотип + короткое имя. Логотип помогает узнаванию (USER_PROFILE: визуальные паттерны для возрастной аудитории). |
| 8 | **Статус** | `status` | ~120px | Бейдж: Активен / На оформлении / Истёк / Расторгнут. |
| + | **Действия** | — | ~80px | Иконка «...» с меню или прямая кнопка «Пролонгировать» если `endDate < 30 дней`. |

**Что убираю и почему:**

- **ИКП агента** — все строки одного агента. Это поле имеет смысл только для куратора, который смотрит чужие портфели. У агента в его «Мои полисы» — мусор.
- **Куратор** — у агента один куратор на всю карьеру. Колонка из 30 строк с одинаковым ФИО — потеря экрана.
- **Дата создания (`createdAt`)** — не вытесняет «Истекает» по важности. Доступна сортировкой и в детальной карточке.
- **№ полиса** — нужен для поиска, но в табличной строке съедает 160px на код вида «РРР-12345678», который агент не читает глазами. Решение: показывать **в детальной карточке** и при наведении/клике на строку (или второй строкой под клиентом мелким шрифтом). Поиск по номеру всё равно работает.
- **VIN** — никогда не нужен в общем списке (16 символов мусора). Только в детали.

**Конкретика по «Объекту страхования»** (зависит от продукта):
- ОСАГО → `Toyota Camry · А123ВС777`
- НС / Антиклещ → `Застрахован: Петрова А. И.`
- Ипотека → `г. Минусинск, ул. Ленина, 5–12`

Чтобы избежать пустоты в этой колонке для не-авто-продуктов — заполняем семантикой продукта.

## Фильтры и поиск

### Раскладка над таблицей (сверху вниз)

```
[Поиск: ФИО, номер полиса, госномер ............ 🔍]   [+ Оформить новый полис]

[ Все (100) ] [ Истекают (12) ] [ Активные (65) ] [ На оформлении (10) ] [ Истёкшие (13) ]

Продукт: [Все ▾]  СК: [Все ▾]  Период: [Этот год ▾]  [Сбросить фильтры]
```

### Что и почему

- **Поиск — крупный, слева сверху**. Поиск важнее «оформить новый» по частоте использования, но «Оформить новый полис» — главный CTA, поэтому справа в primary-кнопке.
- **Пресет-табы (chip-табы)** — это «фильтры в один клик», самое сильное UX-улучшение по сравнению с 1С. Цифры-бейджи дают агенту понять, что **сегодня** надо делать. Это паттерн Сбер Онлайн (вкладки «История / Платежи / Накопления»).
- **Дефолтный таб — «Все»**, не «Истекают». Иначе агент при первом открытии не поймёт, куда делись его полисы. Но таб «Истекают» подсвечен (warning-цветом, если N > 0) — притягивает взгляд.
- **3 фильтра видимы, не за дропдауном** «Фильтры» как в 1С. Спрятанные фильтры = функция, которой не пользуются (см. антипринципы USER_PROFILE: «Не скрывать важные функции в меню»).
- **Поиск по** трём полям (ФИО, номер, госномер) — это уже в `api-contract.yaml`. Можно подсказать в placeholder: `Поиск по клиенту, номеру полиса или госномеру`.
- **Период** — single select с пресетами: Этот месяц / Этот квартал / Этот год / Всё время / Произвольный (открывает date-range picker). Дефолт: **Этот год** (агенту нужны актуальные, не за всю историю).
- **«Сбросить фильтры»** — текстовая ссылка, появляется только когда фильтры применены. Спасение для агентки, которая «нажала чего-то и теперь ничего не показывается» (типовой страх из USER_PROFILE).

### Чего НЕ делаю в фильтрах

- Нет фильтра «Куратор» (у агента один куратор).
- Нет фильтра «ИКП» (это сам агент).
- Нет «Расширенный поиск» как отдельной модалки — фильтры либо видны, либо их нет.
- Нет автокомплита в поиске на V1 (старый ноутбук + 3G — задержки между нажатиями могут раздражать). Поиск с дебаунсом 400ms и явной кнопкой «Найти».

## Действия (на строке + массовые)

### На строке (V1)

- **Клик по строке (любая ячейка кроме колонки «Действия»)** → открывает детальную страницу полиса `/policies/{id}`. Не модалка — отдельный роут, чтобы можно было дать ссылку коллеге/куратору.
- **Колонка «Действия» (всегда последняя, ~80px)**: одна основная кнопка-иконка зависит от статуса:
  - `endDate < 30 дней` и `status = active` → кнопка **«Пролонгировать»** (primary, brand-цвет). Самое важное.
  - `status = pending / processing` → кнопка **«Продолжить оформление»**.
  - Остальные случаи → меню «...» с пунктами «Открыть», «Скачать PDF», «Написать клиенту».
- **Действие «Расторгнуть»** — НЕ на главной строке. Только в детальной странице полиса, с двойным подтверждением (DESIGN_SYSTEM: «двойное подтверждение для опасных»). Расторжение полиса = потеря комиссии агента + проблема для клиента. Спрятать так, чтобы случайно не нажалось.

### Массовые (V2, НЕ в V1)

- Чекбоксы на строках → «Скачать выбранные», «Массовая пролонгация», «Экспорт в Excel».
- В V1 не делаю: добавит сложности, аудитория к этому не привыкла, а сам сценарий редкий (массовая пролонгация = декабрь корпоративных контрактов, не основной поток).

### Что точно НЕ в V1

- Drag & drop строк.
- Inline-редактирование ячеек (расторгать полис кликом в ячейку — ужас для возрастной аудитории).
- Bulk-actions через выделение Shift+Click.
- Действия по правой кнопке мыши (контекстное меню в браузере — не наш паттерн).

## Layout и навигация

### Header (сверху)

Оставляем привычное по 1С, чистим лишнее:

- **Слева:** логотип Agent Academy (всегда ведёт на дашборд — правило USER_PROFILE).
- **Центр:** оставляем пусто или хлебные крошки `Главная / Мои полисы` (DESIGN_SYSTEM требует хлебные крошки).
- **Справа:** ФИО агента + город, иконка-меню (выпадает: Профиль / Настройки / Выйти). Не три отдельных ссылки в линию — это паттерн 1С, заменяем на привычный дропдаун из Госуслуг.

**Цвет header — белый или светло-серый, не navy.** Тёмный header — паттерн 1С/корпоративных порталов 2010-х. Госуслуги и Сбер Онлайн используют светлые headers с тонкой нижней границей.

### Sidebar (слева)

Текущий sidebar — иконки без подписей. Это **прямое нарушение** USER_PROFILE: «иконки **всегда** с подписями».

**Решение:** sidebar шириной 240px, **иконка + текстовая подпись** в одну строку. Сворачивается до 64px только по явному клику на «гамбургер» (и то сомнительно для V1 — лучше всегда развёрнут на десктопе).

**Структура sidebar (V1):**

```
🏠 Главная
📋 Мои полисы          ← активный
🧮 Рассчитать ОСАГО
💰 Комиссии и выплаты
👤 Профиль
─────────────
❓ Поддержка
🎓 Обучение
```

**Что убираю из текущего 1С-sidebar:**

- Отдельные иконки на каждый продукт (машина+, сердце+, дом+) — это **не сценарии**, это типы расчётов. Один пункт «Рассчитать» с выбором продукта внутри. Иначе sidebar разрастётся при добавлении продуктов.
- «Sync» — это техническая операция 1С, агенту не нужна.
- «Конверт» (вероятно письма/уведомления) — переезжает в header (иконка колокольчика справа).
- «Информация» — переезжает в «Поддержку».

**Иконки** — Lucide или Taiga, stroke-width 1.5–2px (DESIGN_SYSTEM), 24px. Цвет — `gray-700` неактивные, `brand-500` фон + белая иконка активная.

### Чат-кнопка

Оставляем в правом нижнем углу (привычный паттерн). НО: судя по PROJECT_BRIEF боль №1 — чат и так в стадии перепиливания. В прототипе V1 чат — заглушка с текстом «Скоро здесь будет чат с поддержкой». Не в скоупе.

## Что я бы НЕ делал в V1

1. **Карточный вид «полисы плитками»** вместо таблицы. Соблазн «современнее», но аудитория работает с таблицами в Excel и 1С каждый день — таблица это родной формат. Карточки = ломка привычек.
2. **Группировка по клиенту** (один клиент = одна раскрывающаяся группа со всеми его полисами). Это путь к разделу «Клиенты» в V2. В V1 — flat-список, проще.
3. **Сложная аналитика на странице** (мини-графики «динамика по месяцам», «структура портфеля по СК»). Это раздел «Дашборд» / «Профиль». На странице полисов — только список и фильтры.
4. **Тёмная тема, кастомизация колонок (показать/скрыть), drag-drop порядка колонок.** Каждая опция — это +1 место, где Светлана 67 может «случайно нажать и сломать». Антипаттерн.
5. **Inline-расчёт пролонгации в модалке поверх таблицы.** Пролонгация = full-screen форма, как при первичном расчёте ОСАГО. Не пытаться запихнуть в модалку — она будет либо обрезанной, либо вылезет за 1366×768.

## Открытые вопросы к пользователю

1. **Какая ментальная единица у агента — «полис» или «сделка»?** Если агент ОСАГО + НС при ДТП = одна сделка с одной комиссией, нам нужна группировка. Если каждый чек отдельный — flat-список. Это влияет на колонку «Комиссия» и группировку строк.
2. **Что означает статус `processing` (есть в моках, но нет в API enum)?** Между `pending` и `active` есть состояние, когда полис в обработке на стороне СК. Нужно решить — отдельный статус или подстатус `pending`? Это влияет на пресет-табы.
3. **Какова реальная частота пролонгаций vs новых оформлений?** Если пролонгации = 70% дохода (что я предполагаю из брифа), пресет «Истекают» — главный таб, и его, возможно, имеет смысл сделать дефолтным. Если 30% — оставляем «Все».
4. **Нужна ли история действий по полису** (кто и когда что менял) на детальной странице? Это влияет на структуру `/policies/{id}` и косвенно — на колонку «Последнее действие» в списке.
5. **Кто видит «комиссию» — только агент или и куратор?** Если куратор тоже работает через эту же страницу (в роли «мои подопечные → их полисы»), у нас два разных view, и тогда колонки «ИКП» и «Куратор» возвращаются — но не в V1 «Мои полисы», а в отдельный экран «Портфель куратора» (вне скоупа V1).
